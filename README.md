# 감사렌즈 (Audit Lens)

> 코스피 전 종목 감사보고서를 **업종 단위로 검색·비교**하고, 모든 답변에 공시 원문 인용과 DART 딥링크를 붙여 돌려주는 회계사용 RAG 서비스

## 시작점 — 유사 산업 비교의 번거로움

이 프로젝트는 감사 실무자 인터뷰에서 나온 한 문장에서 출발했습니다.

> "담당 기업의 감사보고서를 쓰고 첨부되는 주석을 검토할 때, **같은 업종의 다른 회사들은 비슷한 사건이나 거래를 어떻게 기술했는지** 비교해 보는 과정이 번거롭습니다."

핵심감사사항을 어떻게 선정하고 기술했는지, 우발부채나 계속기업 불확실성을 동종 업계는 어떤 수위로 공시했는지 — 이런 비교는 감사 판단의 근거가 되지만, DART에서 회사를 하나씩 열어 주석을 대조하는 일은 오래 걸립니다. 검색 기능도 회사명 위주라 **"조선사들은", "제약사들은"** 처럼 업종 단위로 묶어 보기 어렵습니다.

**감사렌즈는 이 비교를 자연어 한 줄로 대신합니다.** "조선사들의 핵심감사사항을 알려줘"라고 물으면, 업종에 속한 회사들을 찾아 각각의 공시 원문을 근거와 함께 나란히 보여줍니다.

## 유사 산업 비교를 가능하게 한 것 — WICS 업종분류

업종 단위 비교의 전제는 **"이 회사가 무슨 업종인가"를 정확히 아는 것**입니다. 그래서 코스피 808개사 전부에 WICS(FnGuide 산업분류) 체계를 붙였습니다.

- **대분류 26 · 중분류 48 · 소분류 85**의 3단 계층 — "운송장비·부품" 같은 뭉뚱그린 분류가 아니라 "조선 / 자동차부품 / 항공사"까지 구분
- 매핑 출처를 투명하게 관리: 공식 매핑(WMI500) 281사, 원문 딥리서치 보정 125사, 나머지는 추정 후 수동 검증
- 질의에서 업종어를 자동 인식해 검색 범위를 좁히고, 회사 국가·연도·감사/검토까지 사전 필터로 결합

덕분에 "조선사 전부", "제약 업종 중 계속기업 불확실성이 있는 회사"처럼 **업종 × 이슈** 교차 질의가 가능합니다.

## 무엇을 하는가

| 질문 유형 | 처리 방식 | 예시 |
|---|---|---|
| **업종 비교** | WICS 필터 + 사실 조회로 동종 업계를 나란히 | "조선사들의 핵심감사사항 비교" |
| **전수 스크리닝** | 사전 정리한 사실을 SQL로 전수 조회 (생성 없음 → 환각 0) | "계속기업 불확실성이 있는 기업 전부" |
| **단일 기업 조회** | Fact Store 우선 + 본문 보완 검색 | "삼성전자 감사 투입시간과 보수" |
| **개방형 탐색** | 하이브리드 검색(BM25+벡터) → 재랭킹 → 합성 | "이런 사건을 다른 회사는 어떻게 기술했나" |
| **관계 질의** | 특수관계자 지식그래프 순회 | "같은 감사인을 쓰는 기업" |
| **수치·순위** | XBRL 정형재무 SQL 계산 | "부채비율 200% 넘는 기업" |

## 데이터 규모

- 코스피 **808개사** · 감사보고서/검토보고서 **약 3,200건** (3개 사업연도 + 당기 분·반기)
- 검색 청크 **60만 개** · 공시 원본 표 **약 160만 개**
- 사전 정리 사실 **13,523건 · 18종** — 감사의견, 핵심감사사항, 계속기업 불확실성, 감사인 보수·투입시간, 소송·우발부채, 회계정책·추정 변경, 특수관계자 거래, 강조사항, 전기오류 수정, 정정공시 이력 등

## 아키텍처

```
사용자 ─https─ Caddy ─ FastAPI(정적 UI + API + QueryEngine)
                          ├─ OpenSearch  : BM25 + kNN 하이브리드 색인
                          ├─ PostgreSQL  : Fact Store · 정형재무 · 표 스토어 · 대화
                          └─ Claude API  : 이해 / 합성 / 감사자 (역할별 모델 분리)

무인 운영: EventBridge(평일 09시 기동) → 증분 수집·임베딩 → 18시 자동 종료
```

EC2 한 대에 Docker 4컨테이너로 올렸습니다. 마이크로서비스 대신 단일 서버를 택한 것은 **예산(2개월 $150) 안에서 규모에 맞는 복잡도**를 고른 결과이고, 컨테이너 경계는 지켜 나중의 분리를 열어 뒀습니다.

## 핵심 설계 네 가지

### 1. 업종 비교의 근거는 사실을 미리 구조화한 것

"조선사들의 핵심감사사항"에 답하려면 조선 업종 회사들의 핵심감사사항을 **빠짐없이** 모아야 합니다. 벡터 검색은 상위 k개를 주는 표본이라 이 완전성을 보장하지 못합니다. 그래서 적재 시점에 18종의 사실을 구조화해 PostgreSQL에 넣고, 완전성이 필요한 질문은 **WICS 필터 + SQL 전수 조회**로 처리합니다. 이 경로에는 생성이 없으므로 환각도 없습니다.

### 2. 표를 자를 때 맥락을 다시 넣는다 (청킹 v2)

첫 버전은 1,400자 단위로 잘랐는데, 청크 82만 개를 전수 조사해 보니 표가 중간에서 끊기며 **단위와 칼럼 제목이 사라진 사례가 수십만 건**이었습니다. 챗봇이 금액 단위를 답하지 못하던 원인이 모델이 아니라 문서를 자르는 방식이었던 것입니다.

v2에서는 제목·표 같은 **구조를 경계로** 나누고, 긴 표를 조각낼 때는 조각마다 `〔표: 제목〕(단위: 백만원)`과 칼럼 머리행을 다시 주입합니다. 그 결과 표가 온전히 남는 비율이 **80% → 100%**로 오르면서 청크 수는 오히려 **26% 줄었습니다**.

### 3. 원본 표를 방향까지 그대로 보존한다

주석의 표는 공시 원문 XML 구조 그대로 격자로 저장해, 답변에서 "공시 원본 표"로 보여줍니다. 넓고 빈 표가 보기 나빠 행과 열을 바꿔 저장한 적이 있는데, DART 원문과 방향이 달라지자 오히려 신뢰를 잃었습니다. 되돌리고, 대신 격자로 보여주기 어려운 표는 **항목별 목록**으로 분리했습니다. 원문 충실성이 가독성보다 우선한다는 원칙입니다.

### 4. 모델을 믿지 않는 검증 장치(하네스)를 붙였다

- **인용 기계검증** — LLM이 붙인 인용문이 근거 원문에 실제로 존재하는지 문자열로 대조. 실패하면 검증 배지를 주지 않습니다. (감사의 재계산 절차와 같은 발상)
- **답변 감사자** — 생성과 평가를 분리해 별도 모델이 채점하고, 미흡하면 상위 모델로 재작성
- **골든셋 회귀** — 정답을 DART 원문에서 직접 확정한 64문항으로 모든 변경을 검증
- **피드백 루프** — 사용자 반려(👎)를 자동 집계해 평가 문항 후보로 승격 (정답 확정만은 사람이 수행)

## 기술 스택

**백엔드** Python 3.11 · FastAPI · Uvicorn · PostgreSQL 16 · OpenSearch 2.18(nori)
**검색·ML** BGE-M3(1024차원 임베딩) · BGE-reranker-v2-m3 · PyTorch · RRF 융합 · HyDE
**LLM** Claude (Haiku 4.5 / Sonnet / Opus — 역할별 분리, tool use 구조화 출력, 프롬프트 캐싱, Batch API)
**프런트엔드** Next.js 16 · React 19 · TypeScript · Tailwind CSS v4 (정적 export를 FastAPI가 동일 출처 서빙)
**인프라** AWS EC2 · Docker Compose · Caddy(Let's Encrypt) · EventBridge Scheduler · Terraform

## 저장소 구조

```
audit_lens_pipeline/
  src/pipeline/     수집·파싱·임베딩·추출·질의 파이프라인 (단계별 CLI)
    parse_v2.py       구조 우선 청킹 (표 자체완결)
    query.py          QueryEngine — 6경로 라우팅 · 하이브리드 검색 · 하네스
    incremental.py    무인 증분 파이프라인 (멱등)
    tables.py         원본 XML 표 → 격자 스토어
    sync_tables.py    색인 ↔ 표 스토어 정합 동기화
  src/clients/
    wics.py           WICS 업종분류 · 업종어 자동 인식
    opendart.py · opensearch_store.py · postgres.py · claude.py
  ontology/         회계 도메인 온톨로지 (개념·업종·부정어·도메인 규칙)

web-next/           Next.js 프런트엔드
docs/               최초 설계서
```

## 실행

```bash
# 1) 인프라
docker compose -f infra/docker-compose.yml up -d

# 2) 환경변수 (.env.example 참고 — 키는 각자 발급)
cp audit_lens_pipeline/.env.example audit_lens_pipeline/.env

# 3) 파이프라인 (단계별)
python -m src.cli universe      # 코스피 유니버스 스냅샷 + WICS 매핑
python -m src.cli discover      # 대상 보고서 수집
python -m src.cli fetch         # 원본 다운로드
python -m src.cli parse         # 청킹
python -m src.cli embed         # 임베딩 → 색인
python -m src.cli extract       # 사실 추출 → Fact Store
python -m src.cli serve         # API + UI

# 무인 증분 (운영)
python -m src.cli incremental --days 7
```

API 키(OpenDART·KRX·Anthropic)와 데이터셋은 저장소에 포함하지 않습니다.

## 만들면서 배운 것

**데이터를 먼저 본다.** 첫 챗봇의 실패 원인을 모델에서 찾다가, 청크를 전수 조사하고 나서야 문서를 자르는 방식이 문제였음을 알았습니다. 이후로는 무엇을 고치든 데이터부터 확인합니다.

**원문 충실성이 가독성보다 우선한다.** 표를 보기 좋게 돌려 저장했다가, 원문과 어긋나자 되돌렸습니다. 감사 도구는 원문 대조가 주된 사용법이기 때문입니다.

**자동화할 것과 남길 것을 구분한다.** 수집·임베딩·피드백 집계는 무인화했지만, 정답 확정처럼 판단이 필요한 단계는 의도적으로 수동으로 두었습니다.
