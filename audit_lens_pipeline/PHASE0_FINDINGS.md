# Phase 0 실분석 결과 & 설계 수정안

> 표본: 업종 층화 **40사 / 26개 업종 / 연차 117건** (as_of 2026-06-24, v1)
> 데이터 우선 게이트(설계서 §1.4·§3.5)의 산출물. 이 문서가 `parse`/`embed` 설계를 구동한다.
> 원자료: `data/v1/phase0/embedding_spec.json`, `data/v1/meta/disclosures.csv`.

## 1. 파싱 가능성 — 매우 양호
- **parse_failure_rate 0%** (117/117 본문 XML 기계판독). 전 업종에서 이미지표로 인한 추출 실패 없음.
  - (이전 파일럿의 11%는 fetch 버그였고 수정됨 — 실제 포맷 문제 아님.)
- DART 원문 = `<DOCUMENT>` 루트 XML(UTF-8). lxml `recover` 파싱으로 전량 텍스트화.
- **수정안**: 규칙 기반 XML 파서를 주 경로로. **비전 폴백은 희귀 예외(Dead-letter)로만** 둔다.
- ZIP 구조 가변: 본문 1파일 또는 본문+첨부 N파일(`{rcpNo}.xml`, `{rcpNo}_NNNNN.xml`) → **모든 XML 하위문서 처리**.

## 2. 단위·통화 — 설계 핵심, 매우 다양 (320종)
- 지배 스케일: **천원(23.9k) > 백만원(14.7k) > 원(5.1k)**. (건설만 보면 백만원 우세였으나, 전 업종은 천원이 더 흔함 — 단위 가정을 한쪽으로 굳히면 안 됨.)
- **외화 만연**: USD·EUR·JPY·CNY/RMB·VND(동)·HKD·GBP·NOK 등. **한 표에 다(多)통화 선언** 빈번: `천원, USD, JPY`, `천원,백만동,USD,EUR`, `USD,EUR,GBP,NOK`.
- 표기 잡음: 공백/줄바꿈 혼입(`천 \n USD`), 콤마 유무, 특수단위(`US$/Kg`, `USD/Ton`, `만톤(tCO2-eq`, `ADR주`).
- **수정안 (§3.5.2 강화)**:
  1. 단위 출처는 **XBRL `unitRef` 최우선** — 텍스트 선언은 다통화·잡음이 많아 보조.
  2. 텍스트 단위 파서: ① 선언 정규화(공백·줄바꿈 제거) → ② 복합 분해(스케일+통화 다중) → ③ 통화 통제어휘 매핑(KRW/USD/EUR/JPY/CNY/VND/HKD/GBP/NOK + 한글 별칭 달러·유로·위안·엔·동).
  3. 다통화 표는 셀별 통화를 컬럼 컨텍스트/XBRL로 확정. **단위 불명 셀은 적재 금지(격리)**.

## 5. ★운영/아키텍처 수정안 (신규 발견)
- `consolidated_doc_no`는 **dart.fss.or.kr 뷰어 HTML 스크레이핑**이다(OpenDART API 아님).
- **스로틀 없이 157연타 → IP 차단**(`RemoteDisconnected`, 수 분 지속). 전량(808사 ≈ 3,200보고서)이면 확실히 차단된다.
- **수정 완료**: 뷰어 요청에 스로틀(1.5 req/s) + 지수 백오프 재시도.
- **추가 권장**: `discover --no-resolve-dcm`로 빠르게 돌리고, **dcm 해소를 별도의 재개가능(resumable) 단계로 분리** → 차단이 discover 산출물을 오염시키지 않게. (fetch는 OpenDART API라 무관.)

## 6. 미해결 · 검증 대기
- **연결 vs 별도**: 일부 기업은 연결재무제표가 없을 수 있음(자회사 없는 기업 → 별도 감사보고서만). 이번 빈 dcm 52건은 **뷰어 차단 탓**이라 검증 보류. **차단 해제 후 빈 dcm 재해소**로 "연결있음(차단이었음)" vs "별도만" 확정. 별도만이면 **별도 감사보고서 폴백 + `is_consolidated=false` 태깅**.
- 분기/반기 dcmNo: `consolidated_doc_no`에 **`연결검토보고서`** 키워드 추가 필요.
- 주석 번호 헤딩 패턴: `parse` 구현 시 표본으로 정밀화.

## 7. 단일 기업 질의 설계 (신규 — 설계서 보강)
설계서는 "단건" 의도를 §5.7에서 **인식**하도록만 두고, 라우팅·워크스루·UI 예시는 모두 다기업 스크리닝 중심이다(§5.5 라우터에 단일기업 분기 없음, §5.6 워크스루·UI 예시 4개 전부 스크리닝). 단일 기업 질의를 **일급 시나리오**로 추가한다. (비용은 1사 한정이라 온디맨드도 부담 없음.)

### 7.1 하위 유형 → 경로
| 유형 | 예 | 경로 |
|---|---|---|
| 단일 사실 | "현대건설 2024 감사의견?" | **Fact Store** 필터(corp_code+fact_type+year) — 즉답·정확 |
| 회사 프로파일/요약 | "현대건설 감사보고서·KAM 요약" | **벡터검색**을 corp로 스코프 + 구조화 요약 |
| 스키마 밖 1사 | "현대건설 IFRS16 영향?" | **온디맨드 추출**(후보 좁힐 필요 없이 그 1보고서만) |
| 연도 추이 | "현대건설 최근 3년 의견 변화" | **Fact Store** 시계열(corp×year) |

### 7.2 기업명 → corp_code 해소 (단일 기업의 관문)
- 동명이의·계열사("현대건설" vs "현대건설기계"), 우선주("현대건설우"), 사명변경 이력.
- 유니버스(`corp_name`/`stock_code`/`corp_code`) + 온톨로지 별칭으로 후보 매칭. **다중 후보 → 사용자 명확화(disambiguation)**, 단일 → 확정.
- 우선주는 보통주 corp로 통합, 사명변경은 `corp_code` 기준 이력 병합(유니버스 예외규칙 재사용).

### 7.3 회사 프로파일 출력(권장 고정 형식)
한 화면에: 감사의견(연도별) · 핵심감사사항(KAM) · 계속기업 불확실성 · 주요 회계정책/추정 변경 · 특수관계자·우발부채 요지 — 각 항목에 **딥링크 + 섹션경로 + 원문 스니펫**. 대부분 Fact Store에서 즉시 조립.

### 7.4 라우터 반영(§5.5 보강)
**단일 기업 감지(기업명 1개 + '전부/모두' 없음) → 위 4유형으로 세부 라우팅.** 멀티쿼리/온톨로지·인용 기계검증은 스크리닝과 동일하게 적용.

## 8. 스케일 검증 (150사 / 597보고서, 2026-06-25)
업종 층화 150사로 확대해 수집 견고성·스로틀·이미지표·별도비율·비용을 검증.
- **★뷰어 스로틀 차단방지 검증**: `resolve-dcm`이 **597회 뷰어 요청을 차단 없이 완주**(~1.4 req/s, 7분). → 전량 확장 안전.
- **연차 dcm 100%**(447/447): 연결 381 + **별도 66(15%)**, 미해소 0. 별도-only는 실재하는 소수(~15%) → 연결우선·별도폴백·`is_consolidated` 태깅이 필수임이 검증됨.
- **분기 dcm 63%**(94/150): 나머지 56건은 **검토보고서가 분기보고서 본문 내장**(별도 첨부 없음) → rcpNo 딥링크로 폴백(허용). 연차가 핵심이라 영향 작음.
- **parse_failure 0%**(442 연차): 150사에서도 이미지표 추출실패 0 → **비전 폴백은 희귀 예외(Dead-letter)로 격하**, 규칙 기반 XML 파서가 주 경로(설계 단순화).
- **fetch 597/597, 실패 0**: ZIP 검증·정정 회피 견고.
- **비용/시간**: 150사 ~11.5분 · 239MB → **전량 808사 ≈ 1시간 · ~1.3GB** 추정(증분·재개가능).
- **비12월 결산**(3월 결산) 기업 표본에 다수 포함 — discover/FY 로직 정상 처리.
- 단위 744종(외화·복합 더 다양) — §2 단위 전략 유효.

> 결론: **데이터 수집 파이프라인은 스케일에서 검증됨.** 전량 808사 수집은 de-risk됨(언제든 ~1시간 배치). parse 개발에는 150사(597보고서)로 충분.
