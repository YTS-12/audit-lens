## 1. 왜 만들었나 — 유사 산업 비교의 번거로움

이 프로젝트는 감사 실무자 인터뷰에서 나온 한 문장에서 출발했습니다.

> "담당 기업의 감사보고서를 쓰고 첨부되는 주석을 검토할 때, **같은 업종의 다른 회사들은 비슷한 사건이나 거래를 어떻게 기술했는지** 비교해 보는 과정이 번거롭습니다."

핵심감사사항을 어떻게 선정하고 기술했는지, 우발부채나 계속기업 불확실성을 동종 업계는 어떤 수위로 공시했는지 — 이런 비교는 감사 판단의 근거가 됩니다. 하지만 현실은 DART(전자공시시스템)에서 회사를 하나씩 열어 주석을 대조하는 수작업이고, 검색 기능도 회사명 위주라 **"조선사들은", "제약사들은"** 처럼 업종으로 묶어 보기 어렵습니다.

**감사렌즈는 이 비교를 자연어 한 줄로 대신합니다.** "조선사들의 핵심감사사항을 알려줘"라고 물으면, 업종에 속한 회사들을 찾아 각각의 공시 원문을 근거와 함께 나란히 보여줍니다.

## 2. 주요 기능

### 🔍 업종 단위 비교 검색
"조선사들의 핵심감사사항", "제약 업종 중 계속기업 불확실성이 있는 회사"처럼 **업종 × 회계이슈** 를 교차해 질문할 수 있습니다. 이걸 가능하게 하는 것이 아래의 WICS 업종분류입니다.

### 🏭 WICS 3단 업종분류
코스피 808개사 전부에 FnGuide의 WICS 산업분류(**대분류 26 · 중분류 48 · 소분류 85**)를 붙였습니다. "운송장비·부품"처럼 뭉뚱그린 분류가 아니라 **조선 / 자동차부품 / 항공사**까지 구분되며, 질문 속 업종어("조선사")를 자동으로 인식해 검색 범위를 좁힙니다.

### ✅ 근거가 붙은 답변 (환각 방지)
모든 답변에는 **공시 원문 인용문 + DART 원문 딥링크**가 함께 나옵니다. AI가 지어낸 문장이 아니라, 실제 보고서의 어느 위치에서 나왔는지 클릭 한 번으로 확인할 수 있습니다.

### 📊 전수 조회 (누락 없는 스크리닝)
"계속기업 불확실성이 있는 기업 전부"처럼 빠짐없이 훑어야 하는 질문은, 미리 정리해 둔 사실을 SQL로 전수 조회합니다. 검색이 아니라 조회이므로 **누락도 환각도 없습니다.**

### 💬 이어지는 대화 (챗봇)
"삼성전자 감사보수 알려줘" → "작년이랑 비교하면?" → "SK하이닉스는?" 처럼 회사·계정을 다시 쓰지 않아도 맥락을 기억하고 이어서 답합니다.

### 🤖 완전 무인 운영
평일 아침 서버가 스스로 켜져 새 공시를 수집·색인하고 저녁에 종료합니다. 노트북이 꺼져 있어도 데이터가 최신으로 유지됩니다.

### 👍👎 피드백으로 스스로 나아지는 구조
각 근거 카드에 **좋아요/싫어요 버튼**이 있습니다. 사용자가 "이 답은 부실하다"고 반려(👎)하면, 그 질문이 자동으로 집계되어 **향후 골든셋(품질 검증용 정답지) 확장 후보**로 쌓입니다. 즉 사용자의 불만이 다음 품질 개선의 재료가 됩니다. *(정답을 확정해 실제 골든셋에 넣는 마지막 단계만은 사람이 원문을 확인해 수행 — 자세한 이유는 [10장](#10-만들면서-배운-것) 참고)*

### 📮 건의사항 접수
화면 우하단의 **"개선 요청 보내기"** 버튼으로 누구나 기능 제안·오류 신고를 남길 수 있습니다. 접수된 건의는 별도 테이블에 기록되어 운영자가 확인합니다. 실무자의 목소리를 계속 반영하기 위한 창구입니다.

## 3. 핵심 기능 — 어떻게 동작하나

사용자가 질문하면, AI가 먼저 **질문의 의도를 파악**한 뒤 6가지 경로 중 가장 알맞은 곳으로 보냅니다(라우팅). 각 경로는 잘하는 질문이 다릅니다.

| 질문 유형 | 처리 방식 | 이런 질문에 강함 |
|---|---|---|
| **업종 비교** | WICS 업종 필터 + 사실 조회로 동종 업계를 나란히 정렬 | "조선사들의 핵심감사사항 비교" |
| **전수 스크리닝** | 미리 정리한 사실을 SQL로 전수 조회 *(생성 없음 → 환각 0)* | "계속기업 불확실성이 있는 기업 전부" |
| **단일 기업 조회** | 정리 데이터 우선 + 본문 보완 검색 | "삼성전자 감사 투입시간과 보수" |
| **개방형 탐색** | 하이브리드 검색 → 재랭킹 → AI 합성 | "이런 사건을 다른 회사는 어떻게 기술했나" |
| **관계 질의** | 회사·감사인·특수관계자를 연결한 지식그래프 순회 | "같은 감사인을 쓰는 기업" (아래 상세) |
| **수치·순위** | XBRL 정형 재무데이터를 SQL로 계산 | "부채비율 200% 넘는 기업" |

### 관계 질의를 조금 더 자세히

"같은 감사인을 쓰는 기업"처럼 **회사와 회사 사이의 연결**을 묻는 질문은, 미리 만들어 둔 관계 데이터를 그래프처럼 타고 넘어가며 답합니다(지식그래프 순회). 네 가지 관계를 지원합니다.

- **동일 감사인** — "삼성전자와 같은 감사인을 쓰는 기업은?" → 삼성전자의 당기 감사인을 찾고 → 그 감사인이 감사하는 **다른 회사들(피어)**을 모아 → 감사보수까지 함께 보여줍니다.
- **감사인 교체** — "안진에서 삼일로 감사인을 바꾼 회사는?" → 연도별 감사인을 비교해 `전기 감사인 → 당기 감사인`이 바뀐 기업을 찾습니다. 주기적 감사인 지정제로 교체된 회사를 훑을 때 유용합니다.
- **감사인 경유 (3단계 연결)** — "삼성전자 감사인이 감사하는 다른 회사 중 계속기업 불확실성이 있는 곳은?" → 회사 → 감사인 → 그 감사인의 다른 고객사 → 조건 필터, 세 다리를 건너 답합니다.
- **특수관계자망** — "두산에너빌리티의 특수관계자는?" → 주석에서 추출해 둔 특수관계자 연결선(엣지)을 따라 계열·거래 상대를 보여줍니다.

> 이런 질문은 벡터 검색으로는 답하기 어렵습니다. "A와 같은 감사인"을 알려면 A의 감사인을 **정확히 특정**하고 그 감사인의 고객사 명단을 **빠짐없이** 확보해야 하는데, 이는 검색이 아니라 관계 데이터 조회의 영역이기 때문입니다.

그리고 답변을 내보내기 전에 **검증 장치(하네스)** 를 거칩니다. 이것이 감사 도구로서 신뢰의 핵심입니다.

- **인용 기계검증** — AI가 붙인 인용문이 원문에 글자 그대로 있는지 프로그램이 대조합니다. 없으면 "확인됨" 표시를 주지 않습니다. *(감사의 재계산 절차와 같은 발상)*
- **답변 감사자** — 답을 만드는 AI와 채점하는 AI를 분리했습니다. 미흡하면 더 상위 모델로 다시 씁니다.
- **골든셋 회귀 테스트** — 정답을 DART 원문에서 직접 확정한 64개 문항으로, 코드를 바꿀 때마다 품질이 떨어지지 않았는지 자동 검사합니다.
- **피드백 → 골든셋 확장** — 사용자가 👎로 반려한 질문을 매일 자동 집계해 골든셋 확장 후보로 승격합니다. 정답 확정만은 사람이 하되, "무엇을 검증해야 하는가"는 실사용에서 자동으로 발굴됩니다.

## 4. 데이터 — 무엇을, 어디서 가져왔나

### 데이터 출처 (모두 공개된 공식 데이터)

| 출처 | 무엇을 | 용도 |
|---|---|---|
| **OpenDART** (금융감독원 전자공시 Open API) | 감사·검토보고서 원문(XML), 기업개황, XBRL 재무제표 | 서비스의 원천 데이터 |
| **KRX** (한국거래소 정보데이터시스템) | 코스피 상장사 명단·종목코드·업종 | 대상 기업(유니버스) 확정 |
| **FnGuide WICS** | 산업분류 체계·기업별 매핑 | 업종 단위 비교의 기준 |

수집 순서는 이렇습니다. **KRX**에서 코스피 상장사 명단을 받아 **OpenDART**의 기업 고유번호와 종목코드로 연결하고(유니버스 808개사 확정), 각 기업의 감사·검토보고서 원문을 OpenDART에서 내려받습니다.

### 데이터 규모

- **대상**: 코스피 **808개사** · 감사보고서/검토보고서 **약 3,200건** (최근 3개 사업연도 + 당기 분기·반기)
- **검색용 청크**: 약 **60만 개** (보고서를 검색 단위로 자른 것)
- **공시 원본 표**: 약 **160만 개** (주석의 표를 구조 그대로 보관)
- **미리 정리한 사실(Fact)**: **13,523건 · 18종**

### 미리 정리한 사실 18종

전수 조회를 빠르게 하기 위해, 적재 시점에 아래 항목들을 구조화해 데이터베이스에 넣어 둡니다.

```
감사의견 유형 · 핵심감사사항(KAM) · 계속기업 불확실성 · 강조사항 · 내부회계관리제도 검토의견
감사인 보수 · 감사 투입시간 · 비감사용역 계약 · 감사인 변경사유
회계정책 변경 · 회계추정 변경 · 감가상각방법 변경 · 재고자산 평가방법 · 수익인식 정책
소송·우발부채 · 특수관계자 거래 · 전기오류 수정(재작성) · 정정공시 이력
```

이 중 12종은 AI가 추출하고, 표준 서식의 표에서 뽑는 6종(감사시간·보수 등)은 **규칙 기반**으로 추출합니다. 규칙 기반은 AI를 쓰지 않아 비용이 0이고 결과가 항상 같습니다.

### WICS 업종 매핑의 투명성

업종 분류는 출처를 명시해 관리합니다 — 공식 매핑(FnGuide WMI500) **281사**, 원문 딥리서치로 보정한 **125사**, 나머지는 추정 후 검증. 어떤 회사의 업종이 어떤 근거로 정해졌는지 추적할 수 있습니다.

## 5. 청킹 전략 — 문서를 어떻게 잘랐나

> **청킹(chunking)** = 긴 문서를 검색할 수 있는 작은 조각으로 나누는 일. RAG의 성능을 좌우하는 가장 기초적인 단계입니다.

### 첫 번째 시도(v1)의 실패

처음에는 흔한 방식대로 **1,400자 단위로 기계적으로 잘랐습니다.** 그런데 챗봇이 "금액이 얼마냐"는 질문에 단위를 답하지 못하는 문제가 생겼습니다.

원인을 모델에서 찾다가, 만들어진 청크 **82만 개를 전수 조사**했습니다. 결과는 충격적이었습니다.

- 표가 글자 수 경계에서 잘리며 **숫자가 절단된 사례 약 19.6만 건**
- 표의 **칼럼 제목(머리행)이 사라진 청크 약 32.7만 건** — "12,345"만 남고 이게 매출인지 부채인지 알 수 없음
- 표에 딸린 **각주가 분리된 사례 약 13.2만 건**
- 병합된 셀 때문에 **행마다 칼럼 수가 안 맞는 경우 84%**

즉 문제는 AI 모델이 아니라 **문서를 자르는 방식**이었습니다.

### 두 번째 시도(v2) — 구조 우선 + 표 자체완결

v2는 네 가지를 바꿨습니다.

1. **구조를 경계로 자른다** — 글자 수가 아니라 제목(TITLE)·표(TABLE) 같은 문서 구조 단위로 나눕니다. 문장이나 숫자가 중간에 잘리지 않습니다.
2. **표 조각마다 맥락을 다시 넣는다** — 긴 표를 어쩔 수 없이 나눌 때, 조각마다 `〔표: 제목〕(단위: 백만원)` 캡션과 칼럼 머리행을 **반복 주입**합니다. 어느 조각이 검색돼도 그것만으로 해석됩니다.
3. **병합 셀을 펼친다** — 세로로 합쳐진 셀(rowspan)을 각 행에 복원해 칼럼-행 매칭을 맞춥니다.
4. **각주를 표에 붙인다** — 표 뒤의 `(주1)`, `※` 같은 각주를 표 조각에 결합합니다.

**결과**: 표가 온전히 남는 비율이 **80% → 100%**로 올랐고, 짧은 조각을 합치는 처리 덕분에 청크 수는 오히려 **26% 줄었습니다**(82만 → 60만). 정보는 더 온전해지고 양은 줄어든 것입니다.

> 요리에 비유하면, 모델은 요리사이고 청킹은 재료 손질입니다. 감사렌즈의 v2는 **요리사를 바꾸지 않고 손질을 바꿔** 품질을 올린 사례입니다.

## 6. 임베딩·검색 전략 — 어떻게 찾나

> **임베딩(embedding)** = 문장을 숫자 좌표로 바꿔, 뜻이 비슷한 문장은 가까운 자리에 놓는 기술. "매출채권"으로 검색해도 "받을어음"이 걸리는 이유입니다.

### 임베딩 모델

- **BGE-M3** (BAAI) — 문장을 **1,024차원 벡터**로 변환. 한국어에 강하고, 한 번에 긴 텍스트(최대 8,192토큰)를 처리해 앞서 만든 "자체완결 표 청크"를 잘리지 않고 담습니다.
- 모델을 서버에 직접 띄워 씁니다(self-hosted). 60만 청크를 임베딩해도 API 호출 비용이 0입니다.

> 참고로 삼일PwC가 공개한 회계 특화 임베딩 모델도 후보로 검토했지만, 입력 길이 한계(514토큰)가 저희 표 청크(평균 약 1,400자)와 맞지 않아 이번에는 채택하지 못했습니다.

### 검색은 두 방식을 함께 씁니다 (하이브리드)

| 방식 | 무엇을 보나 | 왜 필요한가 |
|---|---|---|
| **BM25 (어휘 검색)** | 단어가 그대로 겹치는가 | 회계는 "정액법 vs 정률법"처럼 **한 글자로 뜻이 갈립니다.** 의미만 보면 위험합니다. |
| **kNN (의미 검색)** | 좌표가 가까운가 | "받을어음"과 "매출채권"처럼 표현이 달라도 같은 뜻을 잡습니다. |

두 검색 결과는 **RRF**(순위 기반 융합)로 합칩니다. 점수 대신 등수를 써서 척도가 다른 두 결과를 공정하게 병합하는 방식입니다.

### 정확도를 높이는 추가 장치

- **멀티쿼리** — AI가 사용자의 한 질문을 비슷한 표현 4~6개로 늘려 각각 검색한 뒤 합칩니다. "실적"을 못 찾던 문제의 해법입니다.
- **HyDE** — 개방형 질문에서는 "정답처럼 생긴 가상의 문단"을 먼저 만들어, 그 좌표로 한 번 더 검색합니다.
- **재랭킹** — 1차로 찾은 후보들을 질문과 한 쌍씩 다시 읽혀(cross-encoder) 순서를 바로잡습니다.

### 사전 필터 — 검색 전에 후보를 좁힌다

검색을 하기 **전에**, 질문에서 드러난 조건으로 대상 문서를 미리 걸러냅니다. AI가 질문을 이해하는 단계에서 아래 조건들을 뽑아내 OpenSearch 검색에 함께 넘깁니다.

| 필터 | 예시 질문 | 걸러내는 것 |
|---|---|---|
| **회사** | "삼성전자의…" | 회사명을 기업 코드로 바꿔 해당 회사 문서만 |
| **업종(WICS)** | "조선사들의…" | 업종에 속한 회사들의 문서만 (WICS 매핑 활용) |
| **연도** | "2024년 기준…" | 해당 사업연도 보고서만 |
| **감사/검토** | "1분기 검토보고서에서…" | 연차(감사) vs 분·반기(검토)를 구분 |
| **연결/별도** | "연결 기준…" | 연결재무제표 vs 별도재무제표 |

이 필터가 **세 가지를 동시에** 개선합니다.

- **정확도** ↑ — 애초에 관련 없는 문서(다른 회사·다른 연도)를 검색 대상에서 빼므로, 엉뚱한 근거가 섞일 여지가 줄어듭니다.
- **속도** ↑ — 60만 청크 전체가 아니라 걸러진 일부만 유사도 계산하므로 응답이 빨라집니다.
- **비용** ↓ — 검색·재랭킹·AI 합성에 넘기는 후보가 적어져 계산량과 토큰 사용이 줄어듭니다.

> 예를 들어 "조선사들의 핵심감사사항"은, WICS에서 조선 업종 회사 코드만 먼저 추린 뒤 그 안에서만 검색합니다. 삼성전자·제약사 문서는 유사도 계산 대상에 아예 오르지 않습니다.

> 단, "이런 사건을 다른 회사는 어떻게 기술했나" 같은 **유사 사례 탐색**에서는 회사·업종 필터를 일부러 풀어, 업종을 넘나드는 사례까지 찾도록 합니다. 질문의 성격에 따라 필터를 조이고 푸는 것도 설계의 일부입니다.

## 9. 저장소 구조 & 실행

### 구조

```
audit_lens_pipeline/          ← 백엔드(파이썬)
  src/pipeline/               데이터 수집→검색까지 단계별 파이프라인 (CLI로 실행)
    build_universe.py           KRX×DART 조인 → 코스피 유니버스 + WICS 매핑
    fetch.py                    OpenDART에서 보고서 원문 다운로드
    parse_v2.py                 구조 우선 청킹 (표 자체완결)
    embed.py                    BGE-M3 임베딩 → OpenSearch 색인
    extract.py / extract_tables.py   사실 18종 추출 → Fact Store
    query.py                    QueryEngine — 6경로 라우팅·하이브리드 검색·하네스
    incremental.py              무인 증분 파이프라인 (멱등)
    tables.py / sync_tables.py  원본 XML 표 → 격자 스토어 + 정합 동기화
  src/clients/                외부 연동 (opendart · krx · wics · opensearch · postgres · claude)
  ontology/                   회계 도메인 온톨로지 (개념·업종어·부정어·도메인 규칙)
  infra/                      Terraform · Docker Compose · Caddyfile
web-next/                     ← 프런트엔드(Next.js + React)
docs/                         최초 설계서
```

### 실행

```bash
# 1) 인프라 기동 (검색엔진 + DB)
docker compose -f infra/docker-compose.yml up -d

# 2) 환경변수 설정 (.env.example 참고 — API 키는 각자 발급)
cp audit_lens_pipeline/.env.example audit_lens_pipeline/.env

# 3) 파이프라인 (데이터 수집 → 서비스까지 단계별)
python -m src.cli universe    # 코스피 유니버스 + WICS 매핑
python -m src.cli discover    # 대상 보고서 목록 수집
python -m src.cli fetch       # 원문 다운로드
python -m src.cli parse       # 청킹
python -m src.cli embed       # 임베딩 → 색인
python -m src.cli extract     # 사실 추출 → Fact Store
python -m src.cli serve       # API + 웹 UI 실행

# 무인 증분 (운영 — 매일 새 공시만 자동 처리)
python -m src.cli incremental --days 7
```

> **API 키**(OpenDART · KRX · Anthropic)와 **데이터셋**(공시 원문·임베딩, 약 5GB)은 저장소에 포함하지 않습니다. `.env.example`에 필요한 키 목록이 있습니다.

## 10. 만들면서 배운 것

**데이터를 먼저 본다.** 첫 챗봇의 실패 원인을 모델에서 찾다가, 청크를 전수 조사하고 나서야 문서를 자르는 방식이 문제였음을 알았습니다. 이후로는 무엇을 고치든 데이터부터 확인합니다.

**AI의 품질은 모델이 아니라 그것을 둘러싼 검증 구조(하네스)가 좌우한다.** 이 프로젝트에서 가장 크게 배운 부분입니다.

- **AI의 자기 신고를 믿지 않는다.** "이 인용은 원문에 있습니다"라는 AI의 말을 그대로 받으면 환각을 거를 수 없습니다. 그래서 인용문이 원문에 실제로 존재하는지를 **AI가 아니라 프로그램(문자열 대조)**으로 확인하게 했습니다. 검증만큼은 확률이 개입하지 않는 결정적 코드로 처리해야 신뢰가 생깁니다. 감사의 재계산(recalculation)과 같은 발상입니다.
- **만드는 역할과 검사하는 역할을 분리한다.** 답을 생성한 AI에게 스스로 채점시키면 후한 점수가 나옵니다. 그래서 답변을 만드는 모델과 채점하는 모델(답변 감사자)을 분리하고, 채점에서 미흡 판정이 나면 더 상위 모델로 다시 쓰게 했습니다. 실제로 감사자가 "수치 표현이 근거와 미세하게 어긋난다"며 자발적으로 반려하고 재작성이 일어난 사례가 있습니다. 감사에서 업무 수행자와 리뷰어를 나누는 것과 같은 통제입니다.
- **품질은 눈이 아니라 게이트로 지킨다.** 코드를 고칠 때마다 사람이 눈으로 확인하면 반드시 놓칩니다. 그래서 정답을 원문에서 확정한 골든셋 64문항을 두고, 모든 변경(모델 교체·재임베딩·기능 추가)이 이 관문을 통과해야만 배포되게 했습니다. 덕분에 모델을 통째로 바꾸는 작업도 하루 만에 안전하게 끝낼 수 있었습니다.
- **검증 항목은 실사용에서 자동으로 자란다.** 무엇을 검증해야 하는지 처음부터 다 알 수는 없습니다. 그래서 사용자가 👎로 반려한 질문을 자동 집계해 골든셋 확장 후보로 쌓이게 했습니다. 다만 "그 질문의 정답이 무엇인가"는 원문 확인이 필요한 판단이라, 마지막 확정만은 사람이 하도록 남겼습니다.

> 정리하면 **에이전트의 품질 = 모델 × 하네스**입니다. 모델은 사서 쓰되, 그것을 감사하는 구조는 전부 직접 설계했습니다. 회계 감사의 언어로 옮기면 내부통제를 설계하고 운영 테스트까지 한 것과 같습니다.

**자동화할 것과 남길 것을 구분한다.** 수집·임베딩·피드백 집계는 무인화했지만, "이 질문의 정답은 무엇인가"를 확정하는 단계처럼 판단이 필요한 일은 의도적으로 사람이 하도록 남겼습니다. 자동화가 잘 될수록, 자동화하면 안 되는 지점을 아는 것이 중요해집니다.
